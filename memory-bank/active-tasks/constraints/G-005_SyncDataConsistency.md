# G-005: 數據同步一致性、日期格式與網路抓取避讓規範

## 1. 日期格式約束 (Date Consistency)
- 強制使用 `YYYYMMDD` 純數字格式進行比對與儲存。
- 避免在資料庫索引中使用斜線 (/)，以降低解析效能損耗。

## 2. 網路抓取避讓機制 (Crawl Evasion & Backoff)
由於專案強烈依賴 Yahoo Finance 等外部 API，為了避免 IP 被暫時性封鎖，必須遵守以下物理約束：

### A. 智慧型同步 (Smart Throttling)
- **併發控制**: `SyncEngine` 全域併發限制為 **2**。
- **令牌速率**: 建議維持 **每秒 2~3 個請求**。
- **人類模擬 (Jitter)**: 處理每隻股票後必須加入 **500ms ~ 1500ms** 的隨機延遲。
- **定期休息**: 每同步 **50 隻股票**，必須強制進入 **30 秒** 的休眠期。

### B. 故障熔斷 (Circuit Breaker)
- **自動冷卻**: 凡偵測到 HTTP Status `0` (RangeError)、`403` 或 `429`，系統必須自動進入 **10 分鐘** 的冷卻期。
- **主動拒絕**: 在冷卻期內，`logFetch` 工具必須主動拒絕發出新的網路請求，避免進一步惡化封鎖狀態。

## 1. 日期格式強制標準
為了確保 SQLite 資料庫與 JavaScript 邏輯比對的一致性與正確排序，系統內所有涉及 K線數據、同步檢查與狀態判定的日期欄位必須遵循以下規範：

- **標準格式**：`YYYYMMDD` (數字型字串，例如 `20240419`)。
- **禁止格式**：帶有斜線 (`/`) 或橫線 (`-`) 的格式（例如 `2024/04/19`）。
- **原因**：字串比對時，`"2024/04/19" > "20240419"` 永遠成立（因為 `/` 的 ASCII 大於 `0`），這會導致同步引擎誤以為現有數據已過期而陷入重複同步的無限迴圈。

## 2. 同步引擎健康檢查邏輯
`SyncEngine` 判定「數據是否為最新 (Fresh)」的邏輯必須嚴格遵循：
- 從資料庫取得該股票的 `MAX(t)`。
- 將其與今日日期（格式必須為 `YYYYMMDD`）進行比較。
- **判定準則**：`last_date >= today` 則視為 `fresh`；否則為 `stale` 或 `missing`。

## 3. 同步跳過機制 (Skip Logic)
在執行 `syncStock` 時，判定是否跳過抓取的條件必須同時滿足：
1. 資料庫實體檢查判定該股票為最新。
2. `localStorage` 中存在今日的成功同步紀錄 (緩存按時間順序記錄系統的重大變更與決策。
